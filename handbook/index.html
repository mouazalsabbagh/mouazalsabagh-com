1Site at a glance owner

In plain English. The site is a set of static HTML pages you edit as files (home, works, about, case studies…) plus a small private back-office in the rec/ folder: a page-maker and a recommendation-letter collector. There's no CMS — you change a file, save, and it's live once uploaded.

Public pages: index.html, works.html / projects.html (the galleries), about.html, services.html, blog.html, contact.html, ai-engine.html, the CV, and 76 project pages under work/. Private (login required): the page manager and the recommendations dashboard.

1.1 Architecture & repo map dev

/                      root pages (index, works, about, contact, …)  -> use ./assets/...
/work/case-study/      36 rich case studies     (case-study-<slug>.html)
/work/project-preview/ 40 project previews       -> these use ../../assets/...
/work/_drafts/         unpublished drafts (noindex, git-ignored)
/assets/   css/ (style.css + design-system.css) · js/ · images/ (WebP) · fonts/ · php/
/rec/      collect.php (recs) · ai.php (AI proxy) · pages.php (page manager)
           config.php (secrets, git-ignored) · templates/page-template.html
/i18n/     i18n.js + strings.json (EN/AR)
/database/ schema + localization workbook (.xlsx)
/handbook/ this document

Path-depth rule

Root pages reference assets as ./assets/…; pages under work/* are two levels deep, so they use ../../assets/…. Get this wrong and images 404.

2Running it locally both

In plain English. Double-clicking an HTML file shows the page, but the recommendation features and page manager need a small local server running first (they're PHP). Start the server, open http://localhost:8000, and everything works.

2.1 The two dev servers dev

Defined in .claude/launch.json:
NameCommandPortUse when
php-backendphp -S localhost:8000 -t .8000Always — serves static pages and runs PHP (recs, page manager).
staticpython3 -m http.server 87658765Static-only quick preview (no PHP).

Start / stop

Use your editor's preview launcher, or run the command directly. php-backend serves the whole site, so prefer it.

2.2 Database (MySQL) dev

The recommendation + pages features need MySQL. Locally it's installed via Homebrew: 
brew services start mysql   # start
brew services stop mysql    # stop when done
mysql -u root               # open a shell
Tables (recommendations, pages) auto-create on first use — no manual SQL.
If a backend page shows a 500 "Database connection failed": MySQL isn't running. Start it, then reload.

↳ DB commands & schema: GUIDES-MANUAL.md §3 · database workbook

3Adding a portfolio page both

In plain English. There are two ways to add a project page. Pick a rich, hand-built page for flagship case studies, or the quick Page Manager form for fast, standard pages. This section covers both.

3.1 Which path should I use? owner

If you want…UseWhy
A flagship, cinematic case study with custom layout, metrics, animationsPath A — hand-authoredMaximum design control; matches the existing 36 case studies.
A standard page fast, no coding, draft → publish, English or ArabicPath B — Page ManagerA form writes the HTML for you from a template. Minutes, not hours.

3.2 Path A — hand-authored rich page dev

1
Copy a canonical template (project-detail-template.html or an existing case study like work/case-study/case-study-balady-app.html) into work/_drafts/<slug>.html.
2
Add <meta name="robots" content="noindex"> while it's a draft so search engines ignore it.
3
Fill content: hero image (WebP), title, metrics, body, and og:image / Twitter meta. Keep the ../../assets/… path depth.
4
Preview at http://localhost:8000/work/_drafts/<slug>.html.
5
When ready, move the file to work/case-study/<slug>.html and paste the card link into works.html / projects.html.
6
Run a broken-asset check (see §10) — expect 0 missing references.
Kebab-case filenames only (case-study-my-project.html). The live Linux host is case-sensitive — MyProject.html works on Mac but 404s in production.

↳ CSS/JS link order & per-page checklist: PORTFOLIO-PAGE-UPDATE-CHECKLIST.md · design tokens: design.md

3.3 Path B — the Page Manager both

The page manager (rec/pages.php) is a private admin tool: fill a form, save a draft or publish, and it writes the live HTML from a template. It shares the same login as the recommendations dashboard.

1
Log in. Go to rec/collect.php?action=admin and enter the admin password.
2
New Page. Open rec/pages.php → + New Page and fill the form.
FieldRequiredNotes
titleYesPage title (heading + <title>).
slugNoURL slug, kebab-case. Auto-derived from the title if left blank.
typeYescase-study · project-preview · page.
langYesen or ar (Arabic renders RTL).
og_imageNoImage path from site root, e.g. assets/images/projects/project1.webp.
body_htmlNoRaw HTML body (admin-trusted, not escaped).
3
Save Draft → the page is written to work/_drafts/<slug>.html with a noindex tag and appears in the list.
4
Publish → the live HTML is written to the folder for its type, the draft file is removed, and you get a card snippet to paste into your gallery.
TypeDraft pathPublished path
case-studywork/_drafts/{slug}.html
(noindex)		work/case-study/{slug}.html
project-previewwork/project-preview/{slug}.html
page{slug}.html (site root)
Security: every create/publish/delete is admin-only and CSRF-protected. body_html is rendered raw — only the trusted admin should use it.

3.4 The template & its tokens dev

The manager renders rec/templates/page-template.html, replacing these tokens (verbatim from the file):
TokenReplaced with
{{lang}}en / ar → <html lang>
{{dir}}ltr / rtl (rtl when lang=ar)
{{robots}}noindex,nofollow meta for drafts; empty when published
{{title}}Page title (HTML-escaped)
{{og_image}}OG image path (defaults to assets/images/projects/project1.webp)
{{base}}'' for root pages, ../../ for work/* types — prefixes every asset path
{{type_label}}"Case Study" / "Project Preview" / "Page" (the kicker)
{{body}}Your body_html, inserted raw
To change the look of all manager-generated pages, edit the template once. (If you add/rename a token, update this table.)

3.5 After publishing owner

Copy the card snippet the manager gives you and paste it into works.html (or projects.html) where the project cards live, then reload the gallery to confirm the link works. That's the one manual step the manager can't do for you.

4English / Arabic (i18n) both

In plain English. The site can flip between English and Arabic, switching the whole layout to right-to-left for Arabic, and it remembers the visitor's choice.

4.1 Using it in a page dev

Tag elements, then load the runtime: 
<a data-i18n="nav.home">Home</a>
<input data-i18n-attr="placeholder:form.email">
<button data-i18n-toggle>العربية</button>   <!-- auto-relabels -->

<script src="i18n/i18n.js" data-i18n-src="i18n/strings.json"></script>
From a work/* page use ../../i18n/i18n.js and data-i18n-src="../../i18n/strings.json".

Language resolution order

?lang=ar → localStorage(site_lang) → <html lang> → default en.

JS API

I18N.setLang('ar') · I18N.toggle() · I18N.t('nav.home') · I18N.lang · event i18n:applied.

4.2 Updating translations dev

Edit the Localization sheet of the workbook, then regenerate: 
python3 i18n/export_i18n.py     # workbook → i18n/strings.json
Commit the new strings.json (24 EN/AR keys today).

↳ Full i18n reference & live demo: i18n/README.md · i18n/demo.html

5The recommendation system both

In plain English. A page where someone can write you a recommendation letter — AI helps draft it — and a private dashboard where you review, tag, and export the submissions.

5.1 How it works dev

rec/collect.php routes (via ?action=): save (public POST JSON → stores submission), admin (login / dashboard), login, logout, status (mark new/reviewed/downloaded), export (CSV). The recommendations table (23 columns) auto-creates on first save.

AI proxy

rec/ai.php forwards a prompt to Anthropic server-side (model claude-sonnet-4-6), keeping the API key off the browser, rate-limited to 10 requests/IP/hour.

5.2 Admin dashboard both

Open rec/collect.php?action=admin, log in, and review submissions. Use the status dropdown to track each one and Export CSV for a backup.

↳ Deployment of the rec system: rec/DEPLOY_GUIDE.html · internals: GUIDES-MANUAL.md §4

6Configuration & secrets dev

In plain English. All passwords and keys live in one private file that is never shared or committed. The rest of the code just reads from it.
Secrets live in rec/config.php (git-ignored). A committed template, rec/config.example.php, shows the shape. Copy it and fill in real values: 
cp rec/config.example.php rec/config.php
KeyPurpose
ANTHROPIC_API_KEYAI proxy key (prefers an env var)
DB_HOST/NAME/USER/PASSMySQL connection
ADMIN_PASS_HASHbcrypt hash of the admin password (never plaintext)
SITE_OWNER_EMAILNotification recipient on new submission
$ALLOWED_ORIGINSDomains allowed to call the save / AI endpoints (CORS)

Generate the admin hash

php -r "echo password_hash('your-admin-password', PASSWORD_DEFAULT), PHP_EOL;"
Login uses password_verify; sessions are hardened (HttpOnly, SameSite=Strict, Secure over HTTPS) and CSRF-protected.

7Deploying to production (cPanel) both

In plain English. Going live = create the database on the host, upload the files, set the private config, and smoke-test. The detailed click-by-click walkthrough is the deploy guide; this is the shape of it.
1
In cPanel → MySQL Databases, create the database + user (matching config.php). Tables auto-create on first use.
2
Create rec/config.php on the server with the rotated key + new passwords. Never commit it.
3
Upload the site, excluding _archive/, .git/, node_modules/.
4
Set PHP ≥ 8 with pdo_mysql (cPanel → Select PHP Version).
5
Smoke-test: submit the form, log into the dashboard, confirm config.php returns 403, and CORS is limited to your domain.
Before any deploy: rotate the live Anthropic API key and the DB/admin passwords. Old values remain in git history — code changes alone do not un-expose them.

↳ Click-by-click: rec/DEPLOY_GUIDE.html · provisioning checklist: NEXT-STEPS.md §A

8Security checklist both

In plain English. The short list that keeps the site and its data safe.
✓Item
☐Secrets only in rec/config.php (git-ignored) — none in tracked PHP.
☐.htaccess denies config.php (returns 403).
☐CORS limited to your real domain (no *).
☐Admin password is a bcrypt hash, distinct from the DB password.
